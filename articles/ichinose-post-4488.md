## 429が降ってきた日

最初に設定した上限はこれだった。


```python
FOLLOW_PER_SESSION = 30   # 1セッションあたりフォロー上限
FOLLOW_PER_DAY     = 100  # 1日フォロー上限
```

「BAN安全圏」のつもりで設けた数字が、そのまま原因になった。


launchd で1日3セッション回すと単純計算で1日90件のフォローが走る。100という上限はほぼ上限ぴったり。しかも各セッションで30件を0.5秒インターバルで一気に処理していたから、15分もかからず30フォローが終わる。BlueSkyのレート制限は公式ドキュメントに数字が出ていない——私が探した範囲では、少なくとも明示的な閾値はどこにも書かれていなかった。だから短時間の集中アクセスがどう判定されるかは完全に手探りだった。


ログに `429` が混じり始めたのは稼働3日目。タグ検索 → いいね → フォローのループを1投稿ごとに0.5秒で回していたので、夜間セッションが集中して叩き続けた結果だった。


## 詰まりポイントを3つ同時に抱えていた

まずログで状況を把握した。


```bash
tail -100 04_Config/logs/bluesky_post_launchd.log | grep -E "429|上限|フォロー"
```

これで「1日上限到達」がいつのセッションで出ているか、`429` がどのタグ処理中に出ているかが一目でわかった。3セッションのうち2セッション目の序盤で上限に達していた。1セッション30件は多すぎる。


### 上限値を経験則で探る

`bluesky_launcher.sh` のガードファイル（`/tmp/bluesky_launcher_guard_$(date +%Y%m%d_%H).lock`）は機能していた。二重起動の問題ではなく、1セッション内の密度が高すぎた。タグ検索 → いいね → フォローを0.5秒で回すのは、場合によっては1分に120リクエスト相当になる。


上限を段階的に落とした。30 → 20 → 15、100 → 80。今のところ `FOLLOW_PER_SESSION = 15` / `FOLLOW_PER_DAY = 80` で429は出ていない。もし再発したらさらに絞る予定。インターバルも0.5秒から1.5秒に延ばした——件数よりも時間密度のほうが効いている可能性がある。


### アンフォロー済みを2週間再フォローし続けていた

設計ミスが一個あった。`bluesky_unfollow_log.json` にアンフォロー済みのDIDを記録していたのに、セッション起動時にそのログを `followed` リストへ読み込んでいなかった。結果として、アンフォローしたアカウントを次のセッションでまた普通にフォローしていた。気づいたのは2週間後。


セッション起動時に `unfollow_log` を読んで `followed` に突っ込む3行を追加した。以来、再フォロー問題は発生していない。ファイルが増えるほど「どこを正とするか」を起動時に統合しないと収拾がつかなくなる。設計当初から考えておくべきだった——追加するのに2週間かかった時点でその失敗は確定している。


### 日次カウントの線形探索が死んでいた

`followed` リストが1万件を超えたあたりから、セッション終盤の処理が目に見えて遅くなっていた。フォロー済み確認に `in` チェックをかけていたが、1万件の配列への線形探索は1件あたり数ミリ秒かかる。積み重なると1セッションの所要時間が伸びる。


1万件超のタイミングで `set()` に圧縮する処理は書いてあったが、それ以前の区間で詰まっていた。日次カウントを `daily_follows` として別に持つように変更し、今日フォローした件数を数えるのに `followed` 全件を舐める処理をなくした。体感できるレベルで速くなった。


## 試してわかったこと

429エラーの原因が「制限値が低すぎる」のではなく「密度が高すぎる」だったのは盲点だった。1日100件という数字は問題なさそうに見えるが、それを短時間に集中させると制限を踏む。「安全のために設けた上限がそのまま原因になった」というオチ。


アンフォローと再フォローの管理は、設計当初から考慮しないと後から収拾がつかなくなる。`followed` リストと `unfollow_log` を別ファイルで持つなら、起動時に必ず統合する——それだけのことを後から足すのに2週間かかった。ファイルが増えるほど「どれが正」かを最初に決めておく必要がある。


BSkyのAPI制限は公式ドキュメントに数字が出ていない。「429が出ない最大値」を経験則で探るしかないのが現状で、1日80件 / 1セッション15件は今のところ問題なく動いている。もしまた429が出たらさらに絞る。
