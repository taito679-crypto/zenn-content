Discord朝通知を作ったら、思ってた以上に「考える時間」が減った話。


## 毎朝5分、同じ確認をやってた

作業を始める前に毎朝やってることがあった。ブログの在庫は何件か、X承認待ちは残ってるか、昨夜エラーが出てないか——これが全部バラバラのファイルとターミナルに散らばってて、把握するだけで5分かかってた。毎日。気が向いたときにしか確認しないと「実はYouTube在庫が3日前から空だった」みたいな事態になる。実際なった。


パイプラインの数が増えるにつれて、この確認コストがじわじわ上がってきた。4月中旬、これをDiscord1通に集約することにした。


## 環境

MacBook Pro M5、Pythonで書いたDiscord Bot（`01_Scripts/discord_bot/bot.py`）をlaunchdで定時実行する構成。設計と実装はClaude Code Sonnet 4.6で進めた。


## 最初の通知が「情報の塊」だった

Claude Sonnet 4.6に「毎朝ダッシュボード送ってくれ」とだけ指示した。実装は10分もかからなかった。最初にDiscordに届いた通知がこれ：


```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
☀️ 04/19 (日) おはようございます
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 今すぐやること
📌 X承認待ちが12件あります → !xapprove で確認
📌 YouTubeの在庫が空です → /youtube-mbti で台本を生成
📌 エラーが発生しています → 下のエラー欄を確認

📦 コンテンツ在庫
🟢 ブルースカイ　39件
🟢 X（エックス）　22件　うち承認済み 12件、未承認 10件
🟢 note　41件
🟢 ブログ　17件
🔴 YouTube　0件

⚠️ 昨夜のエラー（1件）
• KDP：ブラウザ操作がタイムアウト → KDPのページを手動で確認推奨
━━━━━━━

```

データは全部入ってた。ブルースカイ39件、X22件（承認済み12件・未承認10件）、note41件、ブログ17件、YouTube0件。前夜のKDPタイムアウトも1件。見た目は完成に近い。


でも、通知を受け取った瞬間に「今日何をやるか」が決まらなかった。「今すぐやること」が3行並ぶと、どれが本当に急ぎかわからない。通知を見てから、また手動で確認しに行く——という本末転倒な動きをしてしまった。これは設計ミスだった。


## 条件分岐で「空のときだけ」警告を出す

「今すぐやること」は、問題があるときだけ出す設計に変えた。全在庫を毎朝表示するんじゃなく、閾値を超えたときだけ📌を追加する。


```python
items = []
if youtube_count == 0:
    items.append("📌 YouTubeの在庫が空です → /youtube-mbti で台本を生成")
if x_pending > 0:
    items.append(f"📌 X承認待ちが{x_pending}件あります → !xapprove で確認")
if has_errors:
    items.append("📌 エラーが発生しています → 下のエラー欄を確認")

```

これだと、何も問題のない朝は「今すぐやること」欄が空になる。それはそれで「今日は余裕がある」という情報として機能する。YouTubeが0件のときだけ台本生成の通知が出て、X承認待ちがゼロならその行自体が出てこない。


エラーセクションは前夜のログから自動取得してる。4月17日、Brave CDPが落ちてた日にKDPのブラウザ操作がタイムアウトした。ログに残ってた「ブラウザ操作がタイムアウト」という文字列をそのまま通知に乗せて、「KDPのページを手動で確認推奨」まで添えた。エラーが出たことを知らせるだけじゃなく、翌朝に取るべきアクションをセットで書いておくのが地味に効いた。


## 通知は「見た瞬間に動けるか」で設計する

毎朝1通届くだけで、今日の優先度が決まるスピードが変わった——というより、考える必要がなくなった。YouTubeの📌が出てれば台本を書く。出てなければ他のことをやる。それだけ。


エラー通知は「出て終わり」にしないことが身をもってわかった。Brave CDPが落ちてたとき、「タイムアウト発生」とだけ書いてあると翌朝どうすればいいか一瞬迷う。「手動確認推奨」まで書いておくと、通知を見た瞬間に体が動く。エラー検知より次のアクション提示に注力したほうがいい——これが今回いちばん効いた設計変更だった。


SNS投稿の文体については別の気づきもあった。Claudeに生成させた投稿文がどうしても「エンジニアが書いた説明文」になりがちで、実体験の素材を入力に混ぜないと読み手に届かない。作業ログや会話テキストを素材にして書かせると、急に具体性が出る。


## まとめ

Discord朝通知に在庫・承認待ち・エラーを集約したら、毎朝の確認作業が1通で完結するようになった。設計のポイントは「常に全部表示」じゃなく「問題があるときだけ出す」条件分岐で、これで通知を見た瞬間にやることが決まる。エラー通知には次のアクションをセットで書く——これをやらないと通知を受け取った後にまた手動確認が発生する。SNS投稿の文体は、Claudeに任せっきりにせず実体験ログを入力素材にすると具体性が出やすい。
